OnePod： 每日精选最新英文播客，用刷文字的方式听播客

我几乎每天都要听播客。

这是我过去一段时间收听的播客的时长和栏目排行。

但受限于时间和语言，我能听的也只有中文播客，一天连一集都不一定听得完，这导致我会错大量海外的优质内容。

follow builders, not influencers。

在 AI、科技、投资这些方向，里面的一手信息、非共识观点、最锋利的产品判断，绝大多数都藏在海外 builder 的长对话里。

我也精翻过不少优质访谈，精翻有它的极限。再认真的转写也比不上原对话的原汁原味，看别人总结过的东西永远会隔一层。

对话的过程是没有文字描述的那么顺畅的，有时候散，但可以都得到他们怎么定义问题、怎么取舍、怎么判断一个趋势是真是假，全在那一段段没修饰过的对话里。

Onepod 就是专门解决这个问题，

我用这个流程跑了三个月，现在做了有 75 个精选的播客内容。

飞书文档 - 图片：来源内容未完整抓取

它每天会从海量的海外播客里精选一篇做翻译总结。选取的都是最近刚更新的英文播客节目，根据点赞和播放量来进行筛选，获取最优质内容。

这是我最近每天晚上 vibecoding 慢慢的把每个功能都磨出来的小产品。

地址：Onepod.site

它把每天我想听但没时间听的海外播客，自动转成可以快速阅读的中文文章。

每篇内容会包含：

  • 播客基本信息（嘉宾、主题、发布时间、来源地址）

  • 十几个核心观点、几段最精彩的对话片段、

  • YouTube 封面和原始链接、飞书原文入口。

飞书文档 - 图片：来源内容未完整抓取

接下来讲讲这个产品我自己怎么用，另一块是它背后的实现思路，对想要把自己关注的信息，每天定时收到精编内容的人会有帮助。

这个产品怎么用

Onepod 的首页是一个信息流，访问地址 onepod.site

每张卡片对应一期海外播客，配封面、标题、频道名、抓取时间。

我们可以把视频播客当文字信息来刷，看到感兴趣的就点进去。

飞书文档 - 图片：来源内容未完整抓取

详情页包含三部分

飞书文档 - 图片：来源内容未完整抓取

  • 左侧目录：点击可以切换不同的播客

  • 中间内容：显示简介、核心观点、精彩片段

  • 右侧标注：原文位置、原播客信息

最推荐你读的是中间内容的核心观点，大多数时候我打开一篇内容的目的非常具体——快速判断这一期值不值得花一个小时深入。

新观点有没有？有没有可以展开写成文章的角度？有没有一句话让我想回到原视频里再确认一遍？

如果有，再跳回 YouTube 或者飞书原文继续看。

Onepod 它不取代原播客，它只是在海外信息和我之间加了一个筛选层。

以前一集播客听完要一个小时，现在我几分钟就能判断这一期对我有没有价值。

这个产品完全由我和 AI 打造，而现在的信息完全由 Agent 自己选择——翻译——分发，我只需要每天静静的等待它给我推送最值得看的那篇播客内容。

AI 的出现彻底打破了信息获取的门槛，

Agent 则解决了筛选成本太高的问题。

这个产品是怎么自动跑起来的？

一开始 Onepod 是半自动的。

我用几个 skill 对接到 openclaw，openclaw 连到飞书。

这个 skill 给大家开源下，可以拿去修改：

https://github.com/zephyrwang6/myskill/tree/main/podcast-workflow

Agent 每天通过飞书告诉我哪些播客值得看，我从里面选一个，然后让它去抓字幕、翻译、整理、写成文章。跑了一段时间我发现，这件事其实可以完全交出去。

飞书文档 - 图片：来源内容未完整抓取

现在它的整套流程是这样：

1 Agent 每天定时去抓海外播客

2 按播放量、点赞、频道质量自动筛选最值得处理的

3 抓字幕或 transcript

4 调模型改写成中文文章

5 自动存到飞书

6 Onepod 网站从飞书同步内容

7 前端自动更新

我几乎不参与其中任何一步。

这件事给我的启发挺大的。Agent 可以变成一个持续运行的内容系统，每天固定为我生产一份信息资产。

以前我们把社交媒体分为 PGC、UGC，现在有了一种新的形式 AgentGC，你来定义 Agent 的获取信息的范围、写作形式，定时给你推送内容。

这跟"我让 AI 帮我写一篇文章"是两件完全不同尺度的事。工具和系统的差别就在这里。

实现思路

Onepod 实现起来不复杂，它把几件原本独立的事情串了起来。

第一，内容运营总结成 skill 交给 Agent。

核心是一个叫 podcast-workflow 的 skill，它把选题、抓取、转写、翻译、总结、改写、入库这一长串动作串成一条链。

以前每个环节都得手动跑一遍，现在让 Agent 跑一次完事。最开始我还参与选题，后来选题这一步也彻底交给 Agent。

这就是 vibecoding 让我最上头的地方。你不是在写一个一次性的脚本，而是在把自己日常的工作流一件一件外包给 Agent，最后这些 Agent 组合起来就是产品本身。

第二，飞书做内容后台。

文章最终都保存到飞书。飞书的好处是既能编辑也能存储。我平时直接在飞书里看文章、改文章、整理分类，网站则通过飞书开放平台读取这些文档。

对我来说飞书承担了 CMS 的角色，但比 CMS 轻得多——我不用单独搭一套后台、维护登录态、写权限系统。只要飞书里有新文章，网站就同步更新。

第三，Onepod 网站负责展示。

网站部署在 Cloudflare。前端读 Cloudflare KV 里的缓存数据，定时同步 Worker 会定期从飞书拉取最新内容。这样不用每次新增文章都重新部署，新内容几分钟就上线。

第四，产品运维都交给 Agent。

代码在 GitHub，部署在 Cloudflare。日常的开发和维护我基本上让 Agent 来做：改代码、提交、配置 Cloudflare、部署 Worker、修同步逻辑、调页面设计。只要给 Agent 提供 CLI 和 API key，它已经能完成相当一部分产品运维工作

以前一个小产品需要你会前端、后端、部署、运维、内容运营，现在这些事可以一件件交给 Agent，自己只需要判断方向、验收结果、持续调教。

它现在还有什么问题

Onepod 远远谈不上完美。

第一个问题是 AI 写出来的内容还是有点固定。

结构、转折、句式经常落在同一个模板里，读多了能闻到 AI 味。但这事对我来说没那么关键，因为我打开文章的目的是获取观点，不是阅读文学。

第二个问题是模型选择上换过太多次。

最开始我用 Claude Code，写出来的中文最舒服，但成本扛不住。

后来陆续切了几家国产模型，目前主力是 Minimax 的包月套餐——量大、单价低、跑批量任务最划算。

但模型换来换去也带来一个明显的副作用：不同时期生成的文章在格式和文风上不完全统一，读起来能感觉到接缝。

但整体下来，这套系统已经足够帮我每天稳定获取海外播客里最核心的信息。

开源这套思路的价值

Onepod 开源地址：github.com/zephyrwang6/onepod

把 Onepod 开源出来，是因为它更像一个模板。

它的逻辑不限于海外播客。换一套数据源，它可以变成每天追 AI 论文的工具、

  • 每天整理投资访谈的工具、

  • 每天总结某个 YouTube 频道的工具、

  • 每天生成行业简报的工具。

任何"信息源稳定 + 想结构化沉淀"的场景，都能套这一套架构。

Onepod 真正的形态更靠近一个个人信息系统的雏形。

Agent 负责采集和处理，飞书负责存储和编辑，网站负责展示和分发。

三块拼在一起，就是一个能 24 小时自己跑的内容流水线。

再往前走一步，它可以变成一个内容产品。

这是我下一步想做的，用的人多，或者感兴趣，可以扫码加内测群，我会提供上述功能的内测版本，用 Agent 为你定制可共享的信息流。

AI 更重要的作用是让一个人能维护一套原本需要团队才能跑起来的系统。

Onepod 就是这样一个小系统。
