Skill创作手记：我用1块钱把Mycc微信群聊天记录一键转成灵感库

想象力即代码：用 AI 搭建知识库的完整复盘

一天时间，不到 1 块钱，从 11,630 条群聊消息中提炼出可复用的知识体系

那天我翻了一下手机，发现 MYCC 内测群的消息又是“999+”。

其实我刚进这个群时，想法特别简单：就是想能在手机上用 CC。没指望太多。但待久了我发现，这个群经常有非常高质量的讨论——有人问 Claude 怎么调教，有人分享新发现的工具，有人报错求助，有人聊产品思路。慢慢地，我就把它定义为“高质量人类沟通群”，属于重点关注的那种。

问题是我没空二十四小时盯着群看啊。

好在我之前做过两三个聊天记录提取的项目，有点经验。我就想：干脆把这段聊天记录导出来，先做数据清洗，看看能不能构建成一个可用的知识库。

说干就干，导出来一看——好家伙，11,630 条消息，这只是快一个月的群聊记录。

我当时就愣了：这么多消息，得有多少宝藏被淹没了？

你有没有过这种感觉：明明记得群里有人分享过一个特别好的解决方案，但翻了半天就是找不到。或者想起某个讨论很精彩，但只记得零星的几句话，完全串不起来。传统方法是一个个翻记录，或者用关键词搜索——但这太低效了，而且很多有价值的讨论就这样永远沉底了。

所以我决定做一件事：把群聊记录变成一个可以用的知识库。

Part One：三大核心挑战

真正开始做之后，我才发现这件事没那么简单。

挑战一：数据混乱

首先是数据本身就很乱。你想象一下，一个微信群的消息都包含什么？文本消息、引用消息、系统消息、有人发图片、有人发红包、有人“拍一拍”、有人撤回消息……什么都有。

我导出数据后一看，11,630 条消息里混了大量的噪音。

原始数据字段示例：

{

"type": "文本消息 | 引用消息 | 系统消息",

"content": "消息内容",

"senderUsername": "wxid_xxx",

"senderDisplayName": "用户昵称",

"quotedContent": "引用的内容（可选）"

}

我需要解决的问题：

  • 消息类型混杂，怎么区分？

  • sender 信息不统一，怎么标准化？

  • 上下文丢失了，怎么把对话串起来？

  • 噪音数据太多，怎么过滤？

    )

挑战二：内容太浅

处理完数据后，CC 自己生成了 495 个讨论文件，每个文件对应一个话题。

我满怀期待地打开第一个文件——然后就傻眼了。

怎么说呢，就是那种“把聊天记录复制粘贴过来”的感觉。标题也很随意，有的叫“其他话题”，有的干脆是 generic ID。我翻了十几个文件，心里越来越凉：这不就是换了个地方存聊天记录吗？

看完之后我不知道能得到什么有用的东西。这不是我想要的。我要的不是简单的记录，而是可复用的知识。

挑战三：无法使用

第三批问题更致命——拿到这个东西根本用不了。495 个讨论，没有导航，没有索引，想找点东西根本无从下手。这不像一个知识库，更像是一堆散落的文件。

整体就是——乱。

Part Two：四阶段解决方案

面对这些问题，我设计了一套四阶段的处理流程，并用 Agent Team 来执行。

整体架构：Agent Team 协作体系

我规划了 15 个 Agent 的分工体系，实际执行过程中根据需要动态调度——有人负责数据清洗，有人负责话题聚类，有人负责人物画像，有人负责资源整理，有人负责深度聚合。分工明确，并行推进。

Phase 1: 数据清洗（3个 Agent）

├─ Agent 1: 数据清洗工程师 - 过滤噪音

├─ Agent 2: 链接提取专家 - 提取和重构链接

└─ Agent 3: XML 解析专员 - 处理红包/转账/@提及

Phase 2: 对话树 + 微信文章（4个 Agent，可并行）

├─ Agent 4: 对话树架构师 - 构建引用链

├─ Agent 5: 长文本分析师 - 深度分析长内容

├─ Agent 6: 微信链接抓取主管 - 抓取文章全文

└─ Agent 7: 链接验证员 - 验证和修复失效链接

Phase 3: 内容分析（4个 Agent，并行）

├─ Agent 8: 话题聚类专家 - 分类和标签

├─ Agent 9: Sender 分析师 - 分析发言者特征

├─ Agent 10: 质量评分专家 - 评分筛选

└─ Agent 11: 资源整理专家 - 整理工具清单

Phase 4: 结构化输出（3个 Agent，并行）

├─ Agent 12: 灵感卡片设计师 - 生成主题卡片

├─ Agent 13: Sender 汇总页生成器 - 按成员归档

└─ Agent 14: 可视化报表专员 - 生成图表

Phase 5: 优化验收（1个 Agent）

└─ Agent 15: 质量保证专家 - 全程监督

实际执行情况：规划 15 个，重点调度了 Agent 3/4/8/11/13，根据问题动态追加（如人物画像不达标，优化 Agent 4），迭代优化（Agent 8 从 58% 覆盖率优化到 100%）。

阶段一：数据清洗——把噪音变成信号

第一个阶段，我先想办法让 CC 把数据整理干净。

核心方法：引用链重建对话树

这是整个项目的关键。微信的引用消息有一个字段叫 quotedContent，就是“回复 xxx：”后面跟的内容。我利用这个字段，把散落在不同时间点的消息“缝合”成连贯的对话。这样一来，就把时间上分散的消息变成了连贯的问答流。

阶段二：AI 聚类——给内容贴上标签

数据清洗完之后，第二步是让 CC 给内容打标签、做分类。

)

9 大分类体系：

  1. AI Agent 工程 - SubAgent、多 Agent、记忆系统

  2. 工具实践 - 硬件、自动化、Docker

  3. 自动化工作流 - SOP、工作流编排

  4. 踩坑与解决 - 报错解决、避坑指南

  5. 产品思考 - AI 局限、变现、产品形态

  6. 学习方法

  7. 心法与思维

  8. 行业动态

  9. 资源分享

质量评分机制：基于内容长度、引用数量、讨论深度、洞察词、对话线程等维度，总分超过 50 分的讨论标记为“高质量讨论”。这个评分机制帮我筛选出了真正有价值的讨论，而不是把什么都塞进去。

阶段三：人物画像重构——从数据报告到活生生的人

第三步，我让 CC 开始做人物画像。

一开始生成的画像太简单了，就是统计一下发言数量、话题分布。我盯着屏幕看了三秒，然后直接炸了：

“人物画像还是不像人物画像，肯定没根据内容用 AI 再做分析。它没有标签，只有‘在干嘛’？这些都不是核心专长、讨论风格吗？这不是忽悠人吗？重新来一遍。”

这不是人物画像，这是数据报告。 我要的是能让人一眼看出“这个人是谁、擅长什么、说话什么风格”的画像，不是冷冰冰的数字统计。

得，推翻重来。我让 CC 重新设计了一个 10 维度人物画像：

  1. 标签（Tags）

  2. 核心专长

  3. 讨论风格

  4. 思想模型

  5. 方法论

  6. 洞察

  7. 影响力度量

  8. 关系网络

  9. 成长轨迹

  10. 一句话画像

还加了 MBTI 推断——从发言风格推断人格类型。

阶段四：深度聚合——从摘要到可用的知识

最后一个阶段，我又遇到了一个更大的问题。

看着 495 个讨论文件，我突然意识到：这些文件都太浅了。每个 thread 就是简单罗列一下讨论内容，看完之后脑子里什么都没留下。

我试着让 CC 做复盘总结，结果更失望——给我三五个要点，很散，完全不成体系。我盯着屏幕想了很久：我到底要的是什么？

然后我明白了：我要的不是“摘要”，而是可以直接用的知识。

迭代过程：

第一版：简单复盘

  • 我问：“总结这个 thread 的核心观点”

  • CC 给我：三五个要点，很散，不成体系

第二版：发现问题

  • 我意识到：这不是我要的“知识”，这是“摘要”

  • 我调整方向：要求 CC “提炼出方法论、架构图、实践案例”

  • CC 开始尝试结构化，但还是停留在单个 thread 的层面

第三版：跨 Thread 聚合

  • 关键洞察：同一个主题可能散落在多个 thread 里

  • 我让 CC 把所有关于“AI Agent”的讨论聚合起来

  • 提炼出 4 个核心问题，绘制架构图，整理案例

  • 当我看到第一份深度聚合文档时，我愣了三秒——这才是我要的东西！

最后生成了 4 个深度聚合文档：AI-Agent 工程、工具实践、产品思考、踩坑与解决。

Part Three：反思与方法论

事后复盘这个项目，我发现的不是技术问题，而是关于“发现问题和改变”的心法。

三次关键迭代

这次项目经历了三次重要的迭代，每次都是一次认知升级：

第一次受挫：人物画像

看到人物画像的时候，我一眼就看穿了问题——“这不就是数据报告吗？核心专长、讨论风格呢？”我意识到，做产品和做数据的区别在于：数据是死的，人是活的。

第二次调整：标题优化

一开始生成的话题索引里充斥着“其他话题”、“generic ID”这样的垃圾标题，说白了就是 AI 没理解内容在讲什么。这是偷懒的代价——没有用心理解，产出就不会用心。

第三次突破：深度聚合

当我意识到单个讨论文件太浅，不能形成知识体系时，才明白真正的价值不在于记录了多少，而在于提炼出了什么。

)

我的底层方法论：触发-思考-行动

复盘这个项目，我提炼出了一套属于自己的方法论，其实就是我一直以来的思维模式：

一个触发 → 问题出现（看到人物画像不像画像）
一个思考 → 本质是什么？（数据 vs 产品，摘要 vs 知识）
一个行动 → 怎么改？（重新设计框架、跨 thread 聚合）

这不是线性的，而是螺旋上升的：

触发 → 思考 → 行动 → 发现新问题 → 再次触发 → ...

每一次循环，都不是回到原点，而是上升了一个层次。

三个关键洞察

1. 发现问题 > 解决问题

发现问题需要洞察力，解决问题只需要执行力。但大多数人的问题在于——根本发现不了问题，或者假装没看见。

2. 改变的意愿 > 改变的能力

知道怎么改不难，愿意改才是关键。很多人“懒得改”，因为改意味着推翻重来，意味着承认之前的不够好。

3. 细节决定成败，但洞察决定方向

细节打磨很重要（标题、画像、内容），但更重要的洞察是：我要的不是“记录”，而是“可复用的知识”。没有这个洞察，再多的细节打磨也只是在优化错误的方向。

真正该怕的是什么？是把思考全交给 CC，自己却毫无长进。再优秀的模型，如果你自己不思考、不迭代、不成长，用起来也没意义。这三次迭代教会我一件事：直面问题才是最有效的迭代。我不怕发现问题，怕的是发现了不改。

Lulu_OS 知识沉淀方法论

经过整个项目，我总结出了一套可复用的方法论：

1. 数据清洗 → 过滤噪音，标准化

2. 结构化 → 对话树重建（引用链缝合）

3. 标签化 → AI 聚类与分类

4. 质量筛选 → 评分机制

5. 深度聚合 → 体系化知识

6. 可视化 → 架构图 + 学习路径

核心思路是：从噪音中提取价值，从碎片中构建体系。
