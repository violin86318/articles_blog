Codex 与 ChatGPT/Claude 的根本区别

这是整段视频的隐含前提，作者反复强调：

  • ChatGPT / Claude：你上传的文件、生成的文件都存储在云端，是一个"对话沙盒"。
  • Codex：所有输入与产出都直接落在你本地电脑的真实文件系统上，并且 Agent 拥有对本机文件的完整读写权限，还能进一步控制浏览器与操作系统。

这一差别决定了 Codex 不是"更强的 ChatGPT"，而是更接近 Claude Code / Cursor Agent 那一类——面向真实工作流的本地代理——但配了一个对非开发者也友好的图形界面。

七大核心能力解读

能力 1：Full File Access（完整文件访问）

演示场景：把 60 多张收据图片放在 Downloads/receipts to process 下，让 Agent 进行 OCR、分类、汇总，最终生成带 dashboard、分类汇总、付款方式分析的 Excel。

关键点：

  • Agent 通过本机路径直接读取原始文件（而非用户手动逐个上传）。
  • 输出物（xlsx）也保留在本地，可一键 "Open in Folder" 定位。
  • 作者建议：以"项目（Project）= 文件夹"组织工作。每个 Project 就是磁盘上一个真实目录，所有该 Project 内的会话都默认在该目录读写文件，并能用 @文件名 跨会话引用前面产出的文档。

工程含义：这把 Agent 的"工作记忆"显式映射到了文件系统。这是 Claude Code 的设计哲学，但 Codex 把它包装成普通用户也能理解的"项目+聊天"模型。

能力 2：Persistent Memory（持久化记忆）

Codex 区分两类记忆：

| 类型 | 文件 | 谁来写 | 用途 |
| --- | --- | --- | --- |
| 手动记忆 | AGENTS.md | 你（或你让 Agent 写） | 你显式希望 Agent 长期遵守的偏好、规范 |
| 自动记忆 | ~/.codex/memories/ 下的文件 | Agent 自己维护 | Agent 对你工作模式的观察 |

作者的实操建议很务实：

  • 想固化一个偏好（如"以后做 landing page 都用左侧目录布局"），直接对 Agent 说"请记住"，它会把这条写进 AGENTS.md。
  • 不要手动改自动记忆文件，让 Agent 自己维护即可，定期阅读它能反向了解 Agent 对你的画像。

这本质上是 Cursor / Claude Code 早已采用的 AGENTS.md / CLAUDE.md 规范，OpenAI 在 Codex 中做了产品化。

能力 3：Plugins（插件 / 外部连接）

Plugins 是 Codex 与外部 SaaS（Slack、Gmail、Notion 等，作者说当前已超过 100 个）打通的连接层，相当于 MCP / Connectors 的产品化封装。

演示场景：

  1. 装好 Gmail 插件后，让 Agent "翻过去两周邮件，找出所有想找我做付费推广的品牌方，做调研后输出表格"——5 分钟左右产出一份带研究备注的品牌方清单。
  2. 装好 Notion 插件后，让 Agent 从用户的 Notion 中读取以往的高质量长视频脚本，学习作者的写作语气，再为新主题生成一份"听起来就是作者本人写的"草稿。

调用方式上，输入 @ 即可弹出插件菜单，体验对标 Cursor 的 @ 选择器。

能力 4：Skills（技能 / 可复用 SOP）

作者把 Skills 描述为"reusable workflow recipes / SOP"——一份让 Agent 每次遵循的指令文件。它们存放在 plugins/skills/ 下，调用方式是 /skill_name。

两种创建方式：

  1. Prompt-to-Skill（直接生成）：一句话"创建一个叫 X 的 skill，做 Y"。简单，但作者认为效果一般。
  2. 手工迭代后反向萃取（推荐）：先和 Agent 来回打磨，直到产出真的让你满意，再说"我对这个结果满意，请把它沉淀为一个 skill"。Agent 会回看完整对话，把你认可的工作流写成 SKILL.md。

进一步的关键洞察（作者明确提到）：

每一次使用旧 skill，都是迭代它的机会——发现新亮点直接说"以后都这样做"，让 Agent 更新 skill 文件。

这把 skill 当作"活文档"，而不是一次性 prompt。这一思路与 Anthropic 的 Skills 规范完全一致：Skills = 可演化的过程性知识。

Plugin vs Skill 的边界：

  • Plugin = 与外部系统的"连接 + 工具"。
  • Skill = 用这些工具做某件事的"流程指令"。
  • 一个 Skill 可以调用多个 Plugin（例如 brand-deal-researcher 内部调用 Gmail 插件）。

能力 5：GPT Image Access（原生图像生成）

Codex 内置了图像生成能力（GPT-image-2，当前 OpenAI 最强图像模型）。

演示：上传一张毛衣样图，让 Agent 生成 5 张不同人种模特、不同人数（1/1/1/3/5 人）穿着该毛衣的产品图。

工程上值得注意：图像生成在 Codex 中以一个内置 skill (image-gen) 的形式存在，与第三方 skill 平起平坐。这意味着所有"内置能力"也用 Skills 这套抽象统一描述——这是一个干净的设计选择。

能力 6：Browser & Computer Use（浏览器与操作系统级控制）

两个能力以 Plugin 形式提供：@computer use 与 @browser use。

Computer Use 演示：让 Agent 打开本机 Canva 客户端，新建演示文稿，把刚生成的 5 张产品图一图一页放进去。Agent 真实地接管鼠标、操作 Canva。

Browser Use 演示：把生成的 HTML 文件转成一个测验类小应用，然后让 Agent 自己打开浏览器测试这个应用——点击 Start、答题、验证答案对错、检查侧边栏与滚动行为。这本质上是一个自动化 QA 闭环：Agent 既是开发者也是测试者。

这是 Codex 最具颠覆性的能力：人机交互的三件套（键盘、鼠标、屏幕）正式被 Agent 接管，使任何能在 GUI 内完成的工作都进入 Agent 可达范围。

能力 7：Automations（定时与触发式自动化）

任何已经跑通的会话都可以"一句话变定时任务"：

"请每周五早 9 点跑一次这个流程，并更新这张表。"

Agent 会在 Automations 面板中创建一条记录，包含触发时间、所用 Skill、活动状态、上次运行时间等。

这把前面六项能力闭合成一个完整的工作流系统：

Plugins（连接） → Skills（流程） → Automations（调度）

效果上等价于一个为个人量身定制的、由自然语言配置的 Zapier / n8n。

彩蛋：Chronicle（环境感知记忆）

开启后，Codex 会持续对你的屏幕进行截图采样，并把这些视觉上下文喂给 Agent。

演示：作者没有上传任何文件，只说"用 Chronicle 看看我演示文稿里还该加些什么"，Agent 直接基于近期截图，列出了"Codex 超级应用地图"、"全控制环路图"、"prompt 文件示意"等具体补充建议。

作者诚实指出：这功能略有侵入性（always-on 屏幕录制 → 截图）。但其价值也很明确——Agent 第一次拥有了"看到你正在做什么"的环境感知，这是从"被动响应 prompt"走向"主动协作"的关键技术原语。

整体框架的工程评价

把七大能力抽象一下，可以画出 Codex 的能力栈：

[Chronicle 环境感知]   ←— 视觉上下文输入        ↓[File System 全访问] + [Persistent Memory] ←— 持久状态层        ↓[Plugins 外部连接]   ←— 工具层（与 SaaS / 系统对接）        ↓[Skills 可复用流程]  ←— 程序化知识层（SOP）        ↓[Browser / Computer Use] ←— 执行层（GUI 级动作）        ↓[Automations 调度]    ←— 时间维度的触发器

这套结构与业内已经成型的 Agent 设计范式高度吻合（Claude Code、Cursor、Anthropic Skills、MCP 等），但 Codex 的差异在于：

  1. 统一的产品形态：把"开发者工具 + 知识工作工具"合并成一个 GUI App，非工程背景用户也能用。
  2. 以本地文件系统为真实状态：避免了云端沙盒带来的迁移成本和不可见性。
  3. Skills + Plugins 双轴：清晰区分"做什么的流程"和"用什么去做"，这是 ChatGPT 早期 Custom GPT 体系所欠缺的。
  4. Automations 把单次任务变成持续服务：从"AI 助手"过渡到"AI 员工"。

对个人/团队工作流的实际启示

  • 以项目文件夹为单位组织 Agent 工作：天然解决了"Agent 失忆"和"产出散落"两大痛点。
  • 善用 AGENTS.md 把品味和规范固化：这是把个人偏好"编译"为 Agent 行为的最低成本路径。
  • Skill 不要一次写好，要在使用中演化：每一次"我希望以后都这样做"都是一次免费的 SOP 升级。
  • 真正能放大杠杆的是 Automations：定时跑销售线索分析、邮件汇总、内容选题、对账等高频低价值工作，是个人 ROI 最高的切入点。
  • 对 Chronicle 持谨慎态度：在涉及客户数据、机密文档的环境里启用前，先想清楚截图记录的合规边界。

一句话总结

这期视频的真正价值不止在于"教你 7 个功能"，是传递了一个判断：

OpenAI 把 Codex 定位为"通用知识工作 Agent 的桌面 OS"——本地文件是状态，AGENTS.md 是偏好，Plugins 是外设，Skills 是程序，Browser/Computer Use 是执行器，Automations 是 cron，Chronicle 是感知层。当这套抽象被普通用户接受时，AI 才真正从"对话框"走进了"工作台"。
