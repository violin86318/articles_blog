我让 AI 帮我写公众号，踩了无数坑后，做了一个全自动 Skill

想用 AI 自动写公众号？我踩完了所有坑，做了一个从研究到发布的全自动 Skill。这是踩坑全记录和致谢。 

一切的起点

事情要从一个很简单的念头说起。

我本职工作是 IT 项目经理，业余时间折腾跨境电商和各种 AI 工具。最近迷上了 OpenClaw——一个开源的 AI 代理框架，可以让 AI 像真正的助手一样帮你干活：管日程、发消息、写代码、操控浏览器。

用了一段时间后，我有了一个很自然的想法：能不能让 AI 帮我写公众号文章？

不是那种"帮我润色一下"的程度。我想要的是：给一个话题，或者丢一个链接进去，AI 自己研究、写稿、排版、配图、转成微信能发的 HTML，最后直接保存到公众号草稿箱。我只需要最后看一眼，点发布。

听起来不难对吧？

我当时也是这么觉得的。

第一个坑：微信编辑器吞掉了我的样式

第一次尝试的时候，我让 AI 生成了一篇排版精美的 HTML 文章。标题有颜色，代码块有高亮，引用块有左边框和背景色——在浏览器里预览，效果非常好。

然后我把它粘贴到微信公众号编辑器里。

样式全没了。

标题变成了普通文字，代码块失去了高亮，引用块变成了缩进文本。我精心调整的字号、行距、颜色，一个都没留下。

查了半天才搞明白原因：微信公众号编辑器底层用的是 ProseMirror——一个富文本编辑框架。ProseMirror 在处理粘贴内容时，会对 HTML 做"清洗"，把它认为"不安全"或"不规范"的 CSS 全部剥掉。

你可以理解为：微信编辑器有一个隐形的过滤器，你粘贴进去的 HTML 会先被脱一层皮。

最终解决方案：不走编辑器，直接走 API。微信有一个内部接口叫 operate_appmsg，可以绕过编辑器直接把 HTML 写进文章的 content0 字段。这样样式就不会被 ProseMirror 剥离了。

但这只是第一个坑。

第二个坑：微信的 HTML 不是你以为的 HTML

绕过编辑器之后，我以为终于可以用标准的 HTML + CSS 了。

并不能。

• 不能用 div，要用 section。div 在部分客户端上会被忽略

• 不能用 flexbox 和 grid 布局。想做多列？用 table

• 不支持外部 CSS 和 class 选择器。所有样式必须写成 inline style

• 字体必须统一。用 Microsoft YaHei 或微软雅黑

这意味着你得写一种"微信方言的 HTML"——用 section 替代 div，用 table 模拟 flex，每个元素都挂着一长串 inline style。

这些规则没有任何官方文档。全是一个个试出来的。

写了一个 section，在 iOS 上看没问题，在安卓上段落间距消失了。用了一个 line-height: 1.8，在 PC 端看挺好，在手机端行距变成了 1.5。

每一条规则的背后，都是几轮测试和"为什么又不对了"的自我怀疑。

第三个坑：封面图拦截了保存

好不容易搞定了 HTML，文章内容通过 API 成功注入到编辑器——标题有了、正文有了、摘要有了。

点击"保存为草稿"。

弹出了图片选择器。

原来微信公众号的文章要求必须有封面图，没有封面图就不能保存。而这个封面图不是一个简单的图片 URL——你得先通过 filetransfer API 把图片上传到微信的素材库，获得一个 media_id，然后把这个 media_id 填到保存接口的 fileid 字段里。

完整链路：本地图片 → 上传到微信 CDN → 拿到 media_id → 写入草稿保存请求。漏掉任何一步，保存就会被拦截。

第四个坑：工具链在关键时刻崩了

到这一步，我已经把流程理清了，开始寻找现成的工具来串联整条链路。

我找到了宝玉（GitHub: JimLiu）做的一系列 baoyu-skills——他是最早在 AI agent 生态里做"内容生产 skill"的人之一。他的 baoyu-markdown-to-html 可以把 Markdown 转成带主题的 HTML，baoyu-post-to-wechat 可以通过浏览器自动化把文章发到公众号。

问题是，baoyu-markdown-to-html 底层依赖 unified/remark 这套 Markdown 处理生态。在我的运行环境里（Bun 1.3.x），一个叫 unist-util-visit-parents 的模块解析失败了。

就这么一个模块兼容问题，整条链路就断了。

而且这个问题出现的时机特别糟糕——我是在让 AI 子代理执行全流程时发现的。子代理有 10 分钟的执行超时。它花了 7 分钟完成研究、写稿、配图，然后在最后的 HTML 转换环节卡住了。

10 分钟到了。子代理被强制终止。前面所有的工作白费了。

第五个坑：AI 说"成功"，但草稿是空的

最让人哭笑不得的是调试过程中的一次经历。

我通过浏览器 CDP 自动化了整个保存流程。脚本运行完毕，日志里清清楚楚地写着：

[wechat] Body content verified OK.

[wechat] Summary verified OK.

[wechat] Saving as draft...

[wechat] Done.

"verified OK"。太好了。

然后我打开公众号后台一看——正文是空的。字数显示 0。

原因是在粘贴内容之前，有一个图片库的弹窗覆盖了编辑器。脚本以为自己在编辑器里操作，其实是在弹窗的背景层里操作。所有的"验证"都是验证的弹窗层，不是编辑器层。

AI 很认真地完成了所有步骤。只不过是在错误的层上完成的。

所以，我做了一个 Skill

踩完这些坑之后，我决定把所有经验打包成一个系统化的 Skill。

这就是 research-to-wechat——一个端到端的公众号文章生产 Skill。从研究到发布，7 个阶段，全自动。

Phase 1 — 素材收集：给一个话题、URL、或者视频链接，自动抓取并整理成结构化的素材包

Phase 2 — 研究架构：生成研究大纲，包括核心研究问题和 32+ 个分层问题

Phase 3 — 研究整合：把素材整合成带有"已验证 / 推断 / 待确认"三区划分的研究报告

Phase 4 — 正文写作：根据文章类型选择对应的写作框架，生成完整初稿

Phase 5 — 排版与配图：精排 Markdown，生成配图和封面，应用 Pencil 设计模板

Phase 6 — WeChat 交付：转换 HTML，上传图片，保存草稿

Phase 7 — 多平台分发（可选）：一键转小红书卡片、即刻帖子、播客脚本、朋友圈文案

关键设计

Capability Alias：没有硬编码具体工具名。用"能力别名"解耦——wechat-render 这个别名可以解析到不同的工具。工具换了，流程不变。

Fallback Ladder：草稿保存有四级降级策略。L0 走 API，L1 走全自动浏览器，L2 走半自动浏览器（需要扫码登录），L3 给你文件路径手动粘贴。不管环境多残缺，总有一条路能走通。

Pencil 设计模板：内置 10 种排版设计（极简/编辑/杂志/科技/生活/典雅/粗犷/活泼/商务/艺术），每种都有 Light 和 Dark 两个版本，根据文章主题自动匹配。本文使用的是 04 科技（Light）模板。

站在前人的肩膀上

这个 Skill 不是从零开始的。它之所以能存在，是因为有一群人在前面铺了路。

🙏 翔宇 — 深度研究写作方法论

翔宇老师提供了深度研究文章写作的核心思路和方法论框架。

research-to-wechat 的研究架构——四层问题格（基础层、连接层、应用层、前沿层）、结构化研究大纲、证据台账的三区划分（已验证/推断/待确认）——这些设计背后的底层思路，直接来源于翔宇老师对"如何用 AI 写出有深度的文章"这个问题的长期思考和实践。

方法论决定了上限。没有这套思路打底，research-to-wechat 写出来的文章再怎么自动化，也不过是"AI 生成的速食内容"。

🙏 宝玉（JimLiu）— baoyu-skills 生态

宝玉是 baoyu-skills 整个生态的创建者。他做了十几个 skill——从 Markdown 转 HTML、公众号发布、封面图生成、文章配图，到漫画生成、信息图、小红书图片。

这些 skill 定义了"用 AI agent 做内容生产"的基本范式：每个能力一个 skill，skill 之间通过标准接口组合，上层 orchestrator 负责编排。

research-to-wechat 的整个架构思路——capability alias、downstream worker、phase-based pipeline——直接受益于宝玉在 baoyu-skills 里建立的模式。没有这些下游工具，research-to-wechat 就是一个空壳。

🙏 md2wechat — Markdown 转微信 HTML

md2wechat（github.com/geekjourneyx/md2wechat-skill）解决了一个关键问题：把 Markdown 可靠地转成微信兼容的 HTML。

这个工具用 Go 写成，不依赖 Node.js 生态（所以没有 Bun 兼容问题）。它的 AI 模式调用 Claude 来生成样式化的 HTML，内置了多个配色主题（秋日暖光、春日清新、深海静谧）。更重要的是，它支持一键上传图片到微信 CDN 和直接保存草稿。

在其他 HTML 转换工具因为环境问题不可用的时候，md2wechat 成了可靠的 fallback。这也是我在 Skill 里设计 fallback ladder 的直接动机。

现在的样子

v0.4.0 已经发布到了 ClawHub，任何 OpenClaw 用户都可以一键安装：

clawhub install research-to-wechat

📦 GitHub 开源地址：github.com/Fei2-Labs/skill-genie

• 支持的输入：话题、URL、视频链接、已有文章、Markdown 文件

• 支持的框架：深度分析、教程、评论、叙事、趋势报告、创始人信、Newsletter

• 支持的输出：公众号草稿、小红书卡片、即刻帖子、播客脚本、朋友圈文案

整个流程只需要一句话："把这个写成一篇公众号文章"。剩下的，Skill 自己跑。

当然，它还不完美。但它已经从"一个美好的幻想"变成了"一条真正能跑通的流水线"。

每一个坑，都变成了流水线上的一个检查点。每一次失败，都变成了 fallback ladder 上的一级台阶。

这大概就是做工具的意义——把自己踩过的坑铺成别人的路。
